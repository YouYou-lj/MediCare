1. 引言

1.1 编写目的

本文档定义 MediCare 智慧医疗门诊管理系统的软件需求规格，作为系统设计、开发、测试和验收的基准。面向读者包括项目经理、开发工程师、测试工程师和客户代表。

1.2 项目概述

MediCare 是面向基层医疗机构的门诊管理信息系统，覆盖挂号、就诊、处方、药品库存等核心业务流程。系统采用前后端分离架构，前端基于 Vue 3 + Element Plus 构建，后端基于 Spring Boot + Spring Data JPA，数据库使用 MySQL。

1.3 系统范围

系统覆盖以下业务域：

  • 患者信息管理 — 患者档案的增删改查与搜索
  • 基础数据管理 — 科室、医生、排班的基础维护
  • 挂号预约 — 号源查询、挂号、叫号、完成就诊、取消
  • 医生工作站 — 叫号接诊、书写病历
  • 病历管理 — 门诊病历的创建与更新
  • 药品库存 — 药品信息维护、入库/出库、库存预警
  • 处方管理 — 开处方、取药、作废处方
  • 系统管理 — 用户管理、修改密码

1.4 术语与缩写

术语含义
VO (View Object)视图对象，用于前端展示的数据结构，包含关联查询的冗余字段
DTO (Data Transfer Object)数据传输对象，用于前后端交互的请求/响应结构
CRUDCreate / Read / Update / Delete，增删改查
BCrypt密码哈希算法，自带盐值，用于安全存储密码
SessionHTTP 会话，服务端存储用户登录状态
乐观锁通过版本号/条件判断实现并发控制，避免数据冲突
原子扣减使用 SQL WHERE 条件（如 WHERE stock >= qty）保证并发安全

2. 总体描述

2.1 系统架构

系统采用前后端分离的四层架构：

层次技术栈职责
前端展示层Vue 3 + Element Plus + Vite页面渲染、交互逻辑、路由管理、状态管理
接口层Spring Boot REST Controller请求分发、参数校验、权限控制(@RequireRole)
业务逻辑层Spring Service + @Transactional业务规则校验、事务编排、库存安全扣减
数据访问层Spring Data JPA + MySQL实体持久化、原生SQL投影查询、原子操作

认证机制：Session + Cookie；权限控制：AOP 切面 + 自定义 @RequireRole 注解。

2.2 用户角色

角色角色代码职责范围
管理员admin全部功能，包括基础数据维护、用户管理、挂号操作
医生doctor叫号接诊、病历书写、开处方、查看患者和药品
药师pharmacist药品入库/出库、库存预警、取药/作废处方

项目要求
操作系统macOS / Linux / Windows
JDK17+
Node.js18+

浏览器Chrome / Edge / Safari 最新版
后端端口8080

2.4 设计约束

  • 前后端必须分离部署，通过 RESTful API 通信
  • 认证采用 Session 机制，前端需设置 withCredentials 携带 Cookie
  • 库存扣减必须使用原子操作（WHERE stock >= qty），防止超卖
  • 原生 SQL 投影查询必须使用接口投影（Interface Projection），不能用 @Data class
  • 密码存储必须使用 BCrypt 加密

3. 功能需求

3.1 认证与授权

3.1.1 功能描述

提供用户登录、登出、获取当前登录用户信息的功能。系统采用 Session + Cookie 机制进行身份认证，通过 AOP 切面校验角色权限。

REQ-AUTH-01用户可通过用户名和密码登录系统。功能需求
REQ-AUTH-02登录成功后，系统将用户信息存入 HttpSession，密码字段置空后返回给前端。安全需求
REQ-AUTH-03密码验证兼容明文密码和 BCrypt 加密密码（迁移过渡期）。功能需求
REQ-AUTH-04被禁用的账号（status ≠ 1）无法登录，提示"账号已被禁用"。功能需求
REQ-AUTH-05未登录用户访问受保护接口时返回 401 状态码。安全需求
REQ-AUTH-06角色权限由 @RequireRole 注解声明，RoleCheckAspect 切面在方法执行前校验；角色不匹配时抛出 BusinessException。安全需求
REQ-AUTH-07用户登出时销毁 Session。功能需求

3.5 挂号预约

3.5.1 功能描述

挂号是门诊核心流程的起点，连接患者、排班和医生工作站。支持号源查询、挂号、叫号、完成就诊和取消挂号。

REQ-REG-01挂号时需选择患者和排班（号源），系统在事务内原子扣减号源（WHERE remain_slots > 0），防止超卖。功能需求
REQ-REG-02号源不足时提示"号源不足，请选择其他号源"。功能需求
REQ-REG-03挂号时从 schedule 复制 doctorId 到 registration，保障历史数据医生归属稳定。功能需求
REQ-REG-04挂号时自动分配序号（同排班内递增）。功能需求
REQ-REG-05挂号状态流转：候诊(0) → 就诊中(1) → 已完成(2)。功能需求
REQ-REG-06叫号操作将状态从候诊(0)改为就诊中(1)，仅允许对候诊状态操作。功能需求
REQ-REG-07完成就诊操作将状态从就诊中(1)改为已完成(2)。功能需求
REQ-REG-08取消挂号时将状态改为已取消(3)，同时在事务内回增号源。功能需求
REQ-REG-09已完成的挂号不能取消。功能需求

REQ-WS-01医生选择自己后，系统加载该医生今日候诊和就诊中的患者列表。功能需求
REQ-WS-02叫号后，患者状态变为"就诊中"，医生可进行诊断。功能需求
REQ-WS-03完成就诊后，患者状态变为"已完成"。功能需求
REQ-WS-04工作站集成病历编辑和处方编辑功能，在就诊中状态下可操作。功能需求

3.7 病历管理

3.7.1 功能描述

病历与挂号为 1:1 关系，一个挂号只能创建一份病历。病历包含主诉、现病史、既往史、体格检查、诊断和医嘱。

REQ-MR-01病历字段包括：挂号ID、患者ID、医生ID、主诉、现病史、既往史、体格检查、诊断、医嘱。数据字段
REQ-MR-02同一挂号不允许重复创建病历。功能需求
REQ-MR-03支持按患者ID或挂号ID筛选查询。功能需求
REQ-MR-04更新病历时仅允许修改医学字段（主诉/现病史/既往史/体格检查/诊断/医嘱）。功能需求
REQ-MR-05支持按挂号ID直接查询对应病历。功能需求

3.8 药品库存管理

3.8.1 功能描述

管理药品信息和库存，支持入库、出库和库存预警。药品名称+规格组合唯一。

REQ-MED-01药品信息包括：名称、规格、单位、单价、库存、安全库存、拼音码、生产厂家、状态。数据字段
REQ-MED-02创建/更新药品时校验名称+规格组合唯一性。功能需求
REQ-MED-03入库操作在事务内增加库存 + 记录库存日志（含批次号、有效期、操作人、备注）。功能需求
REQ-MED-04出库操作在事务内安全扣减库存（WHERE stock >= qty 防超卖）+ 记录库存日志。功能需求
REQ-MED-05库存不足时出库失败，提示"库存不足"。功能需求
REQ-MED-06库存预警功能查询库存低于安全存量的药品。功能需求
REQ-MED-07支持按关键词（名称/拼音码/厂家）搜索药品。功能需求
REQ-MED-08库存变动日志记录每次入库/出库操作，包含：药品ID、类型(in/out)、数量、批次号、有效期、操作人、备注。功能需求

3.9 处方管理

3.9.1 功能描述

处方是连接病历和药品库存的关键环节。医生开处方时自动扣减库存，药师负责取药或作废。

REQ-RX-01处方信息包括：病历ID、患者ID、医生ID、总金额、状态。数据字段
REQ-RX-02处方明细包括：药品ID、数量、用法、用量、单价、小计金额。数据字段
REQ-RX-03开立处方为事务操作：保存处方主表 + 逐条安全扣减库存 + 记录库存日志 + 保存明细项。任一步骤失败则整体回滚。功能需求
REQ-RX-04总金额由明细项小计自动汇总，不需要前端传入。功能需求
REQ-RX-05处方状态：待缴费(0) → 已缴费(1) → 已取药(2)；可从待缴费/已缴费作废为已取消(3)。功能需求
REQ-RX-06取药操作将处方状态改为已取药(2)，由药师执行。功能需求
REQ-RX-07作废处方为事务操作：逐条回滚库存 + 记录库存日志 + 更新处方状态。功能需求
REQ-RX-08已取药的处方不能作废。功能需求
REQ-RX-09处方详情查询包含明细项及关联的药品名称、规格、单位（批量查询避免 N+1）。功能需求
